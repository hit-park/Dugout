# AI 라인업 품질 검증 키트 (하위 프로젝트 A) — 설계

- 날짜: 2026-06-16
- 상태: 설계 확정 (구현 계획 대기)
- 관련 엔진 스펙: [`2026-06-03-sabermetrics-lineup-design.md`](./2026-06-03-sabermetrics-lineup-design.md)

## 배경 — 포트폴리오 피벗

선수 데이터 수급 문제로 정식 런칭 대신 **포트폴리오 목적**으로 전환한다. 두 목표:

1. (풀스택 어필) iOS 개발자가 AI를 활용해 서버·웹·안드로이드까지 구축하고 실서버에 배포.
2. (깊이 앵커) 세이버매트릭스 기반 AI 라인업 추천이 **실제로 좋은 추천을 내는지 검증**.

전체는 4개 하위 프로젝트로 분해되며 각자 spec → plan → 구현 사이클을 가진다. 진행 순서는 **A → B → D 코어, C는 옵션**.

| 하위 프로젝트 | 내용 | 목표 |
|---|---|---|
| **A (이 문서)** | AI 품질 검증 키트 | 깊이 앵커 (목표 2) |
| B | 웹 프론트엔드 (라인업 추천 화면, api 풀 파이프라인 증명) | 목표 1 |
| C (옵션) | 안드로이드 동일 화면 | 목표 1 |
| D | 무료 티어 배포 | 목표 1 |

## A의 목적과 비목적

**목적**: 세이버매트릭스 타순 엔진(`dugout-ai/app/services/batting_order.py`)이
(1) The Book lite 슬로팅을 규칙대로 수행하고
(2) shrinkage(k=50)로 작은 표본 노이즈를 억제한다는 것을,
**알려진 특성을 심은 데이터로 반증 가능하게 증명**하고, 같은 데이터로 **돌아가는 데모**까지 보여준다.

면접에서 "이 AI가 좋은 추천을 낸다는 걸 어떻게 아느냐"에 **자동화된 테스트 + 결정적 재현**으로 답하는 것이 1차 산출물.

**비목적 (YAGNI)**:
- api 풀 파이프라인(팀/경기/출석/인증 시딩 → POST → 추천)은 A에서 다루지 않는다. → B(웹)에서 증명.
- wOBA/마르코프 등 정통 지표 도입(엔진 스펙에서 이미 미채택 결정).
- 크롤러의 다중 시즌/페이지네이션/증분 수집. 단발 스냅샷만.

## 결정 요약

| 결정 | 선택 | 이유 |
|---|---|---|
| 검증 형태 | 엔진 직결 pytest + 엔진 직결 데모(A2) | 빠른 반증 + 가벼운 데모. 풀 파이프라인은 B로 이관 |
| 합성 생성기 | 아키타입 × 표본크기 2축 | The Book 배치와 shrinkage 동작을 둘 다 반증 가능 |
| gameone 크롤러 | 범용(club_idx 파라미터), 단 40837부터 검증 | (a)의 상위집합. 상대 스카우팅 데모 확장 여지 |
| 데이터 합류 | 두 소스 → `PlateAppearance` 카운트 공통 출구 | 합성·실데이터가 동일 포맷으로 엔진 진입 |
| 단언 강도 | 혼합 — 정확 슬롯 + shrinkage 분산 | 두 헤드라인 주장 모두 증거화 |

## 아키텍처

전부 `dugout-ai`(Python)에 위치. 엔진과 pytest가 거기 있으므로 모듈 경계가 자연스럽다.

```
[아키타입 생성기] ─┐
                  ├─→ [stat-line] ─→ [PA 전개기] ─→ AttendeeProfile raw 카운트
[gameone 크롤러] ─┘                                        │
                                          ┌────────────────┴────────────────┐
                              (검증) batting_order.order() 직결      (데모) 엔진 직결 + 결과 출력
```

### 컴포넌트

각 컴포넌트는 하나의 책임을 갖고 독립적으로 테스트된다.

1. **아키타입 생성기** — `app/tooling/synthetic.py`
   - 입력: 아키타입 enum + 표본 크기(PA 수) + seed.
   - 아키타입별 타석 결과 확률 분포를 정의(예: 순수출루형 = 높은 볼넷률·낮은 장타, 거포형 = 낮은 출루·높은 홈런/2루타, 종합형 = 둘 다 높음, 평범형 = 팀 평균 근처).
   - 출력: `AttendeeProfile`의 raw 카운트(singles…in_play_outs)를 채운 stat-line.
   - **결정성**: 주입된 seed로만 `random.Random(seed)` 사용. 전역 난수 금지.
   - 의존: 없음(순수 함수).

2. **gameone 크롤러** — `app/tooling/gameone.py`
   - 입력: `club_idx`(기본 40837).
   - `https://www.gameone.kr/club/info/ranking/hitter?club_idx=<idx>` GET → 정적 HTML 테이블 파싱.
   - 컬럼 매핑: 총안타/1루타/2루타/3루타/홈런/볼넷/사구/희타/희비/타수/타석 → stat-line.
     - 단타(1루타)는 페이지에 직접 있음. in_play_outs는 `(타수 - 안타 - 삼진)`으로 도출, reached_on_errors는 0으로 둔다(페이지에 없음 → ROE는 아웃 취급과 일관).
   - **PII 마스킹**: 선수명은 적재 시점에 마스킹(`박**`) 또는 등번호 치환. 원본 실명을 픽스처/커밋/로그에 남기지 않는다.
   - 의존: requests + BeautifulSoup(또는 표준 html.parser).

3. **stat-line → PA 전개기** — `app/tooling/expand.py`
   - 입력: stat-line(집계 카운트).
   - 출력: 같은 카운트를 가진 `AttendeeProfile`(엔진이 카운트를 직접 받으므로 개별 PA 객체 N개로 풀 필요 없이 카운트 그대로 전달). "전개"는 향후 api 경로(B)에서 PlateAppearance N건으로 풀 때 재사용할 공통 인터페이스로 설계.
   - 두 데이터 소스의 합류점.

4. **검증 테스트** — `tests/test_lineup_validation.py` (pytest)
   - 생성기로 만든 선발 9명을 `batting_order.order()`에 직접 먹여 단언.

5. **데모 스크립트** — `scripts/demo_lineup.py`
   - 소스 선택(합성 시나리오 or `--club-idx`) → stat-line → 엔진 직결 → 추천 타순 + reason 문자열 출력(콘솔/마크다운).
   - api·인증 불필요(엔진 직결, A2).

## 검증 테스트가 단언하는 것 (혼합)

데이터는 모두 결정적(고정 seed). 각 단언은 독립 테스트 함수.

### (1) The Book 정확 슬롯 — 큰 표본(예: PA 200)
명확히 분리된 아키타입 9명을 생성:
- 종합형 1명 → **정확히 2번**
- 순수출루형 1명(종합형보다 ISO 낮음) → **정확히 1번**
- 거포형 1명(최고 ISO) → **정확히 4번**
- 6~9번은 adj_OBP 내림차순인지 확인.

생성기는 동점이 안 나도록 아키타입 간 레이트 간격을 충분히 둔다(큰 표본 + 뚜렷한 분리).

### (2) shrinkage 동작 — 표본 축(PA 5 / 200)
**메커니즘 레벨**에서 단언한다. 슬롯 순서가 아니라 보정 지표(`_adjusted`)를 직접 본다 — The Book가 최고타자를 2번에 놓는 로직과 엉키지 않도록(슬롯 순서 vs 스킬 순서 비교는 좋은 추천일수록 순위상관이 깎이므로 부적절).

- **(a) 평균 수축**: 극단 아키타입(예: 거포) 선수를 동일 팀 맥락(평범형 8명) 안에서 PA 5 vs 200으로 생성. `abs(adj_iso(PA5) - team_iso) < abs(adj_iso(PA200) - team_iso)` — 작은 표본일수록 팀 평균에 더 가까움.
- **(b) 분산 압축**: 9명의 adj_OBP spread(max-min)가 `spread(PA5) < spread(PA200)` — 작은 표본일수록 선수 간 차이가 평균으로 눌림.
- 해석: 작은 표본에선 극단값을 신뢰하지 않아 추천이 운에 덜 휘둘리고, 큰 표본에선 또렷해진다. 데모에서는 "5타석 라인업 vs 200타석 라인업"을 나란히 출력해 이 차이를 눈으로 보여준다.

### (3) 콜드 스타트
모든 카운트 0인 9명 → `order()`가 `None` 반환(폴백 신호) 확인.

### (4) gameone 실데이터 스모크
크롤러로 40837을 긁어 9명 이상이면 `order()`가 예외 없이 완주하고 1~9 슬롯이 유일하게 배정됨을 확인(품질이 아니라 파이프라인 무결성).

## 에러 핸들링 (dugout-ai 규칙 준수)

- 크롤러: 네트워크 실패·HTTP 4xx/5xx·파싱 실패를 각각 구분해 `AIException`(code 명시)으로 변환. `print` 금지, logger 사용.
- 빈 결과(선수 0명)와 네트워크 실패를 다른 코드로 구분.
- 생성기: 잘못된 아키타입/음수 표본은 `ValueError`(내부 도구이므로 HTTPException 불필요)로 즉시 거절.

## 보안·개인정보

- gameone 실명은 적재 즉시 마스킹/등번호 치환. 테스트 픽스처·커밋·로그·데모 출력에 실명 금지.
- 합성 데이터는 `user_id=1..N` 합성 ID만(루트 가드 준수).
- 외부 URL은 코드에 상수로 두되 크리덴셜·토큰 없음(공개 페이지).

## 성공 기준

- `pytest dugout-ai/tests/test_lineup_validation.py`가 결정적으로 통과(재실행 시 동일).
- (1)(2)(3) 단언이 모두 의미 있는 형태로 존재하고, 일부러 엔진 규칙을 깨면(예: take 순서 변경) 빨갛게 실패함(반증 가능성 입증).
- `python dugout-ai/scripts/demo_lineup.py`가 합성 시나리오와 `--club-idx 40837` 모두에서 추천 타순 + reason을 출력.
- 크롤링 산출물에 실명이 평문으로 남지 않음.

## 미해결/후속

- B(웹)에서 api 풀 파이프라인(팀/경기/출석/인증 시딩 → PlateAppearance POST → 추천)을 증명. PA 전개기 인터페이스를 그때 재사용.
- D(배포) 스택은 A·B 확정 후 결정. 라인업 기능만이면 PostGIS·Redis 없이 무료 티어 선택지가 넓어짐.
